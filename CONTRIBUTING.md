# Contributing to Public Sector Playbooks

Thanks for helping improve this project. This repository is used in an educational setting, so participation is intentionally broader than "write code." Questions, corrections, examples, documentation fixes, and careful suggestions all count.

## Quick decision guide

Use an **issue** when you want to start a conversation or report something:

- Ask a question
- Report a confusing instruction, typo, or broken link
- Suggest a new resource, lab, playbook, or feature
- Describe a problem you found but are not ready to fix yourself

Use a **pull request** when you already have a specific change ready:

- Edited Markdown
- Added or corrected a template
- Improved a lab, setup step, or checklist
- Added a sanitized example or citation

If you are not sure, open an issue first. That is normal and welcome.

## Good student contributions

Helpful first contributions include:

- clarify a checklist or response step
- suggest a new playbook or tabletop scenario
- add a citation to CISA, NIST, FBI/DOJ, or a state/local source
- improve an executive-facing explanation

## Safety and privacy rules

Do not include sensitive incident details, private network diagrams, credentials, or agency-specific data.

Also avoid uploading:

- secrets, API keys, passwords, tokens, or private keys
- private screenshots or logs that expose names, accounts, networks, or organizations
- copied commercial training content unless redistribution is clearly allowed
- AI-generated text that you have not checked for accuracy

If you think you found a security or privacy problem in the repo, follow `SECURITY.md` instead of opening a public issue.

## Issue workflow

1. Pick the closest issue template.
2. Explain what you noticed or what you want to improve.
3. Include the file/page path if possible.
4. Add screenshots only when they do not expose sensitive information.
5. If this is for class, say so; that helps reviewers understand the context.

## Pull request workflow

1. Fork the repository.
2. Create a branch with a short name, for example `docs/fix-lab-typo` or `feature/new-checklist`.
3. Make a small, focused change.
4. Check your work:
   - Markdown renders cleanly
   - links work
   - instructions are accurate
   - no sensitive data is included
5. Open a pull request and fill out the template.

## Review expectations

Review is part of the learning process. Maintainers may ask clarifying questions or suggest edits. That does not mean the contribution failed; it means the project is being kept accurate, safe, and useful.

## Writing style

- Write for students and practitioners.
- Use clear headings and short steps.
- Prefer plain language over jargon.
- Cite primary or official sources when possible.
- Keep examples reproducible and sanitized.
